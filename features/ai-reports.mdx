Kairo's report service gives Team Leaders on-demand narrative analysis of their coders' performance. Reports are generated by a Python FastAPI service that calls the Groq LLM (`llama-3.3-70b-versatile`), structures the output as JSON, and persists it to the `ai_reports` table.

## Report targets

The `report_target_enum` PostgreSQL type defines the three scopes a report can cover:

```sql
CREATE TYPE report_target_enum AS ENUM ('coder', 'clan', 'cohort');
```

## Requesting a clan report

The Node.js backend calls the Python report service at `POST /generate-report`. The request body must match the `ReportRequest` Pydantic model:

```json
{
  "clan": "Turing",
  "tl_id": 42,
  "total_coders": 18,
  "average_score": 74.3,
  "high_risk_count": 3,
  "top_struggling_topics": ["async/await", "SQL joins", "REST API design"],
  "soft_skills_summary": {
    "autonomy": 3.2,
    "time_management": 2.8,
    "problem_solving": 3.5,
    "communication": 3.9,
    "teamwork": 4.1
  }
}
```

### Request fields

| Field | Type | Required | Description |
|-------|------|----------|-------------|
| `clan` | string | Yes | Clan name (e.g., `Turing`, `Tesla`, `McCarthy`) |
| `tl_id` | integer | Yes | ID of the requesting TL; stored as `target_id` in `ai_reports` |
| `total_coders` | integer | Yes | Total coders in the clan at time of generation |
| `average_score` | float | Yes | Clan-wide average Moodle score |
| `high_risk_count` | integer | Yes | Number of coders with unresolved high or critical risk flags |
| `top_struggling_topics` | array of strings | No | Topics most commonly flagged in `coder_struggling_topics` |
| `soft_skills_summary` | object | No | Clan averages for all five soft skill dimensions |

<Note>
Report generation involves a live call to the Groq LLM. Generation typically completes in 3–8 seconds depending on network latency and model load. The endpoint will return a 500 error if the model returns malformed JSON — this is caught and logged to `ai_generation_log`.
</Note>

## Report response

A successful `POST /generate-report` call returns:

```json
{
  "success": true,
  "report": {
    "report_title": "Clan Turing — Performance Report",
    "generated_date": "March 14, 2026",
    "risk_level": "medium",
    "summary": "Clan Turing is showing moderate performance with 74.3/100 average...",
    "risks": "Three coders are flagged at high risk, primarily driven by low autonomy scores...",
    "recommendations": "Prioritize 1:1 sessions with flagged coders and reinforce async/await fundamentals...",
    "action_items": [
      "Schedule individual check-ins with the 3 high-risk coders this week",
      "Assign a guided async/await activity to the full clan",
      "Review time management resources with coders scoring below 3 on that dimension"
    ]
  }
}
```

## The `ai_reports` table

Every successfully generated report is immediately persisted to `ai_reports`:

```sql
CREATE TABLE ai_reports (
  id            SERIAL PRIMARY KEY,
  target_type   report_target_enum NOT NULL,  -- 'coder', 'clan', 'cohort'
  target_id     INT NOT NULL,
  summary_text  TEXT NOT NULL,
  risk_level    risk_level_enum,
  recommendations TEXT,
  generated_at  TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
  viewed_by_tl  BOOLEAN DEFAULT FALSE
);
```

| Field | Description |
|-------|-------------|
| `target_type` | Scope of the report: `coder`, `clan`, or `cohort` |
| `target_id` | The ID of the target entity (TL ID for clan reports, coder ID for coder reports) |
| `summary_text` | The LLM-generated summary of current clan or coder state |
| `risk_level` | Overall risk assessment returned by the LLM: `low`, `medium`, or `high` |
| `recommendations` | The LLM-generated action recommendations paragraph |
| `generated_at` | Timestamp of when the report was created |
| `viewed_by_tl` | `false` by default; set to `true` when the TL opens the report |

## Report read tracking

The `viewed_by_tl` flag tracks whether a TL has viewed each report. When a TL opens a report in the dashboard, the platform updates this field to `true`. This allows the system to:

- Highlight unread reports with a visual indicator in the dashboard
- Surface only unread reports in notification summaries
- Give program coordinators visibility into TL engagement with AI insights

## PDF report export

In addition to the JSON AI report, TLs can download a formatted PDF for a clan via `GET /generate-pdf/{clan}`. This endpoint does not call the LLM — it fetches coder profiles, soft skill scores, and Moodle progress directly from Supabase and builds a PDF using `fpdf2`. The PDF includes:

- Clan overview (total coders, average score, diagnostic completion rate)
- Individual coder profiles with soft skill scores and learning style
- A visual header with the Kairo brand color (`#6D28D9`)

The file is returned as a streaming response with the filename `clan_{clan}_report.pdf`.

## Generation audit log

Every call to the LLM — whether successful or not — is recorded in `ai_generation_log` with the input payload, output payload, model name, and a `success` boolean. This table provides a complete audit trail for all AI activity on the platform.
