Kairo's database runs on **PostgreSQL 14+** hosted on **Supabase**. It stores all user data, academic progress, AI-generated plans, exercises, feedback, and analytics.

Row Level Security (RLS) is enabled on every table. Policies enforce that coders can only read and modify their own data, while Team Leaders (TLs) have read access to all records in their clan. The Python AI microservice uses the Supabase **service role key** which bypasses RLS entirely.

## Tables

| Table | Description |
|-------|-------------|
| `users` | All platform users: coders and Team Leaders |
| `soft_skills_assessment` | Onboarding diagnostic results per coder (autonomy, time management, etc.) |
| `user_profiles` | Extended profile data: phone, bio, GitHub, LinkedIn, portfolio |
| `modules` | Bootcamp curriculum modules (Python, SQL, JavaScript, etc.) |
| `topics` | Individual topics within each module |
| `moodle_progress` | Academic progress per coder per module, synced from Moodle |
| `coder_struggling_topics` | N:M relationship tracking which topics a coder struggles with |
| `complementary_plans` | AI-generated personalized learning plans (the "6 Cards") |
| `plan_activities` | Individual daily activities within a plan |
| `activity_progress` | Tracks which activities each coder has completed |
| `evidence_submissions` | Files and links uploaded by coders as activity evidence |
| `tl_feedback` | Messages and feedback from TLs to coders, with read notifications |
| `risk_flags` | Automatic risk alerts for coders with low scores or soft skills |
| `ai_reports` | Executive performance reports generated by the AI for TLs |
| `ai_generation_log` | Complete audit log of every AI generation call |

## Row Level Security

RLS is enabled on all 14 tables. The general access pattern is:

- **Coders** can `SELECT`, `INSERT`, and `UPDATE` only their own rows (enforced via `auth.uid()`).
- **Team Leaders** can `SELECT` all rows; `INSERT` and `UPDATE` on tables they manage (plans, activities, feedback).
- **Service role** (used by the Python AI microservice) bypasses all RLS policies.

For the full policy definitions see `rls_policies.sql`. Key policies include:

- `users_tl_sees_all` — TLs can read all user records
- `plans_tl_create` — only TLs can insert new plans
- `feedback_coder_mark_read` — coders can mark their own feedback as read
- `ai_log_tl_only` — the AI generation log is readable only by TLs

## Connection

All services connect to Supabase via the official `supabase-js` (Node.js) and `supabase-py` (Python) clients.

```
SUPABASE_URL=https://<project>.supabase.co
SUPABASE_SERVICE_KEY=<service-role-key>   # AI microservice — bypasses RLS
SUPABASE_ANON_KEY=<anon-key>              # Frontend and Node.js gateway
```
